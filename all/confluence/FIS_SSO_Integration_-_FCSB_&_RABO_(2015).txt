## Method Details

  

### General

  1. In both methods listed below, logic is pretty much same
  2. OLB make a server-to-server call to obtain a unique session ID generated by InfoIMAGE
  3. OLB produce a 302 redirect response to end-user's browser carrying the session ID
  4. InfoIMAGE validate that session ID and allow access to site.

  

### SSOv7

  1. This is a regular HTML form POST with following parameters
    * CIF Number
    * Email Address
    * Current Date
    * and more
  2. key-value pair structured, more keys can be added, but has to remain in flat structure.
  3. reference to [Fiserv_SSO_v7.pdf](https://i3dev.easyredmine.com/attachments/2757/Fiserv_SSO_v7.pdf?t=1437696753 "Download file") for details.

  

### SSO - custom XML

  1. This is a HTML form POST that expects XML content in POST body.
  2. Sample XML specification as follow:  

<BusinessSingleSignOnRequest>  
    <CIID>00001234</CIID>  
    < Password >!2345abcde</ Password >  
    <Customer>  
        <UserId>business_johnDoe111</UserId>  
        <UserType>admin</UserType>  
        <UserName>John Doe</UserName>  
        <UserEmail>john_doe@[infoimageinc.com](http://infoimageinc.com)</UserEmail>  
        <Account>  
            <AccountNumber>123123123</AccountNumber>  
            <AccountType>DD</AccountType>  
        </Account>  
        <Account>  
            <AccountNumber>321213212</AccountNumber>  
            <AccountType>LN</AccountType>  
        </Account>  
    <Customer>

</BusinessSingleSignOnRequest>

    
          
    

  3. reference to [InfoImage SSO Specifications.docx](https://i3dev.easyredmine.com/attachments/2758/InfoImage%20SSO%20Specifications.docx?t=1437696956 "Download file") for details
